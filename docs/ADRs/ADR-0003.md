# ADR-0003 — DetectedEvent And CoachingMoment Are Separate Concepts

## Status

Accepted

## Context

Chess analysis and user-facing explanations serve different purposes.

A detector should identify a chess concept.

A coaching system should explain the concept in a way that helps the user improve.

Combining these responsibilities would tightly couple analysis and presentation.

## Decision

The system separates analysis objects from coaching objects.

Analysis flow:

DetectedEvent  
+ EngineAssessment  
→ VerifiedEvent

Coaching flow:

VerifiedEvent  
→ CoachingMoment

DetectedEvents must never contain:

- Coaching language
- Advice
- Recommendations
- Educational explanations

Example:

Good:

event_type = "hanging_piece_created"

Bad:

"You should have defended your bishop."

## Consequences

### Positive

- Separation of concerns
- Easier testing
- Multiple coaching styles can be built on top of the same analysis
- Future UI changes do not affect detectors

### Negative

- Additional translation layer required
